# Отчет по исследованию моделей и инфраструктуры (Задание 1)

**Контекст задачи:** Разработка RAG-бота для корпоративной базы знаний QuantumForge Software. 
**Ограничения:** Наличие конфиденциальной документации, регулярные SOC 2 аудиты, текущая инфраструктура на базе AWS (EKS + RDS).

## 1. Сравнение LLM-моделей

| Критерий | Локальные LLM (Hugging Face: Llama 3 8B, Qwen) | Облачные API (OpenAI, YandexGPT) |
| :--- | :--- | :--- |
| **Качество ответов** | Достаточное для RAG при качественном промптинге и CoT. | SOTA-качество, глубокое понимание контекста, низкий уровень галлюцинаций. |
| **Скорость (TTFT / TPS)** | Зависит от железа. Без GPU инференс неприемлемо медленный. | Высокая и предсказуемая, масштабируется на стороне провайдера. |
| **Стоимость (TCO)** | Высокий CapEx (закупка/аренда GPU). Нулевой OpEx за токены. | Оплата за токены (OpEx). При 18k документах и активном использовании может быстро расти. |
| **Развертывание** | Сложное (требуется настройка vLLM/Ollama, оркестрация). | Простое (REST API). |
| **Безопасность (SOC 2)** | **Идеально.** Данные не покидают закрытый контур компании. | Риски утечки. Требуются строгие NDA и Enterprise-тарифы с zero-data-retention. |

## 2. Сравнение моделей эмбеддингов

| Критерий | Локальные (Sentence-Transformers, BGE-m3) | Облачные (OpenAI text-embedding-3-small) |
| :--- | :--- | :--- |
| **Скорость индексации** | Высокая, особенно при использовании батчинга на CPU/GPU. | Ограничена rate limits API и сетевыми задержками. |
| **Качество поиска** | Отличное. Модели семейства BGE часто превосходят API-решения на задачах retrieval. | Высокое "из коробки", большое окно контекста. |
| **Стоимость** | Бесплатно. Крутится на существующих CPU-мощностях. | Оплата за каждый токен. Переиндексация базы требует повторных затрат. |

## 3. Сравнение векторных баз

| Критерий | FAISS | ChromaDB |
| :--- | :--- | :--- |
| **Тип решения** | Библиотека для поиска (C++), построение HNSW графов. | Полноценная векторная СУБД (DuckDB/SQLite под капотом). |
| **Скорость** | Максимально высокая производительность поиска. | Немного уступает FAISS на сыром поиске, но не критично для 18k документов. |
| **Внедрение и поддержка** | Требует написания собственной обертки для хранения метаданных и сериализации индексов на диск. | Прозрачная интеграция с LangChain. Сама управляет эмбеддингами, метаданными и персистенцией. |
| **Стоимость владения** | Минимальная (in-memory). | Низкая (легко поднимается в Docker-контейнере). |

*Примечание: для текущего MVP и быстрой проверки гипотез целесообразно использовать FAISS, так как он не требует подъема отдельного сервиса и легко упаковывается в минимальный Docker-образ.*

## 4. Рекомендуемые конфигурации инфраструктуры

**Вариант 1: MVP / Пилотный проект (Текущая реализация)**
* **Оборудование:** Локальная машина / базовая виртуалка (8 CPU cores, 16GB RAM).
* **Стек:** FAISS (in-memory) + локальные эмбеддинги (`all-MiniLM-L6-v2`) + внешняя LLM по API (OpenAI/YandexGPT).
* **Плюсы:** Максимальная скорость разработки, нулевые затраты на GPU, возможность быстро отладить RAG-пайплайн.

**Вариант 2: On-Premise Production (Полная изоляция для SOC 2)**
* **Оборудование:** Выделенный сервер (16 CPU cores, 32GB RAM, 1x GPU уровня RTX 4090 / A10G).
* **Стек:** ChromaDB + локальные эмбеддинги (BGE-m3) + локальная LLM (Llama 3 8B через vLLM).
* **Плюсы:** Полный контроль над данными, отсутствие OpEx за генерацию, соблюдение всех security-политик.

**Вариант 3: Cloud-Native Production (Интеграция в контур QuantumForge)**
* **Оборудование:** AWS EKS кластер (Node pools с GPU-инстансами `g5.xlarge` для LLM-воркеров).
* **Стек:** Qdrant/Milvus (как StatefulSet) + локальные эмбеддинги + локальная LLM.
* **Плюсы:** Нативная интеграция в текущую инфраструктуру компании, отказоустойчивость, автомасштабирование подов при пиковых нагрузках саппорта.

### Итоговый выбор для текущего проекта 

Для реализации бота в рамках данного спринта я выбираю **Вариант 1 (MVP)** с использованием **встроенного FAISS, локальных эмбеддингов (HuggingFace) и API внешней LLM**. 

**Аргументация:**
1. FAISS позволяет собрать минимально рабочий Docker-контейнер без необходимости оркестрации нескольких сервисов (что идеально для сдачи работы).
2. Локальные эмбеддинги позволяют бесплатно экспериментировать с чанкованием и переиндексацией.
3. Внешняя LLM по API снимает необходимость в GPU для локальной разработки, позволяя сфокусироваться на архитектуре самого RAG (настройке поиска, Few-shot и CoT промптинга), а не на тюнинге инференса моделей. В дальнейшем, при переводе в Production (AWS EKS), модуль генерации ответа можно легко переключить на локальную модель без изменения логики поиска.

# Задание 2. Подготовка базы знаний

**Цель:** Создать уникальную базу знаний, которая не пересекается с данными обучения внешних LLM, гарантируя честную работу механизма RAG.

## Выбор предметной области

В качестве исходного материала выбрана вселенная **Star Wars** (starwars.fandom.com). Это широко известная вселенная, присутствующая в датасетах обучения всех современных LLM. Поэтому прямая подача статей про Дарта Вейдера или Люка Скайуокера привела бы к тому, что модель отвечала бы «по памяти», без обращения к векторному индексу.

**Решение:** полная обфускация ключевых терминов — персонажей, планет, фракций, технологий и рас — на вымышленные названия. Таким образом, текст остаётся логически связным и читаемым, но для LLM он становится полностью незнакомым.

## Словарь замен

Словарь замен (`scripts/terms_map.json`) содержит **102 термина** по пяти категориям:

| Категория | Примеры замен |
| :--- | :--- |
| **Персонажи** | Luke Skywalker → Kael Dawnstrider, Darth Vader → Xarn Velgor, Princess Leia → Sera Valdris |
| **Локации** | Tatooine → Solara Prime, Coruscant → Nexara, Dagobah → Myreth |
| **Фракции** | Galactic Empire → Dominion Ascendancy, Rebel Alliance → Liberation Front, Jedi Order → Luminary Order |
| **Технологии** | Death Star → Void Core, lightsaber → lumiblade, the Force → Synth Flux, hyperspace → voidspace |
| **Расы** | Human → Terran, Wookiee → Grukkan, Ewok → Sylvan |

Словарь построен так, чтобы сохранить стиль и логику мира: технологии обфусцированы как технологии, фракции — как названия организаций, имена персонажей звучат естественно.

## Логика обфускации

Скрипт `scripts/obfuscate.py` реализует алгоритм longest-match-first:

1. **Сортировка по убыванию длины** — составные фразы (например, «Darth Vader») заменяются раньше одиночных слов («Vader»), исключая частичные замены.
2. **Сохранение регистра** — re.sub с callback подстраивает регистр первой буквы под оригинал (например, «Darth Vader» → «Xarn Velgor», «darth vader» → «xarn velgor»).
3. **Плоский словарь** — все категории из `terms_map.json` объединяются в единый словарь `{original: replacement}` перед компиляцией регулярного выражения.

## Источники текстов

Вместо парсинга HTML с fandom.com (что создало бы легаси-зависимость от структуры сайта и краулеров) текстовый материал сгенерирован вручную в стиле энциклопедии. Это гарантирует:
- Чистый Markdown без HTML-шума и рекламы.
- Фокус на ключевых сущностях вселенной (персонажи, локации, фракции, технологии).
- Полный контроль над содержимым и его логической структурой.

## Результат

В папке `knowledge_base/` создано **38 документов** (`.md`), каждый посвящён одной сущности:

- Персонажи: Kael Dawnstrider, Xarn Velgor, Sera Valdris, Dax Orren, Eron Vael, Zoru, Overlord Malgrath, Grukk, Koda Veth и др.
- Локации: Solara Prime, Nexara, Glacius, Myreth, Sylvara и др.
- Фракции: Dominion Ascendancy, Liberation Front, Luminary Order, Void Disciples, Iron Vanguard и др.
- Технологии: Void Core, lumiblade, Synth Flux, voidspace, Crosswing, Stellar Hawk.
- Концепты: Clone Wars, Order 66, hyperspace travel, Stormtroopers.

**Проверка целостности:** при чтении обфусцированных текстов сохраняется логическая связность повествования. Например, в статье о Xarn Velgor (Darth Vader) по-прежнему описывается его падение из-за страха потерять Lyra Amaris (Padme Amidala), а в статье о Synth Flux (the Force) — баланс между Lumen (light side) и Void (dark side).

Таким образом, создана полностью оригинальная база знаний, которую LLM невозможно «угадать» по памяти — она вынуждена обращаться к векторному индексу.

# Задание 3. Создание векторного индекса базы знаний

**Цель:** Преобразовать обфусцированную базу знаний в векторный индекс FAISS для семантического поиска.

## Выбор эмбеддинг-модели

| Параметр | Значение |
| :--- | :--- |
| **Модель** | `sentence-transformers/all-MiniLM-L6-v2` |
| **Репозиторий** | https://huggingface.co/sentence-transformers/all-MiniLM-L6-v2 |
| **Размер эмбеддингов** | 384 (dense float32) |
| **Макс. длина последовательности** | 256 токенов (WordPiece) |

**Аргументация выбора:** all-MiniLM-L6-v2 — оптимальный баланс качества и скорости для MVP. На бенчмарках STS она показывает ~81.5% Spearman correlation, что достаточно для retrieval по корпоративной документации. Модель лёгкая (~80 МБ), быстро загружается на CPU и не требует GPU.

## Чанкование текстов

Для разбиения использован `RecursiveCharacterTextSplitter` с иерархическими разделителями: `["\n\n", "\n", ". ", " ", ""]`. Это сохраняет семантическую целостность энциклопедических текстов лучше, чем жёсткое разбиение по символам.

| Параметр | Значение | Обоснование |
| :--- | :--- | :--- |
| **chunk_size** | 800 символов | ~200 WordPiece-токенов с запасом до лимита модели (256 токенов). Достаточно для абзаца энциклопедии, но не настолько большой, чтобы «размыть» семантику. |
| **chunk_overlap** | 150 символов | ~19% перекрытия. Гарантирует, что факты на границе чанков не потеряются — критично для точного retrieval. |

**Результат чанкования:** 38 документов → **76 чанков** (в среднем 2 чанка на статью).

## Построение и сохранение индекса

Использована нативная интеграция `FAISS` из `langchain_community.vectorstores`:

1. Каждый чанк обогащён метаданными `{"source": <имя_файла>}` — для будущего цитирования источников.
2. Эмбеддинги вычислены батчами локально на CPU.
3. Индекс сохранён в папку `vector_index/`:
   - `index.faiss` — собственно векторный индекс FAISS.
   - `index.pkl` — сериализованные документы и метаданные.

**Время генерации:** ~2.1 секунды (76 чанков × 384-мерный эмбеддинг на CPU).

## Дымовой тест (проверка качества)

Запрос: *"Кто такой Xarn Velgor и как он погиб?"*

Результаты поиска (top-3):
1. `darth_vader.md` — релевантный чанк о броне и флоте Xarn Velgor.
2. `endor.md` — чанк о гибели Overlord Malgrath и искуплении Xarn Velgor.
3. `mustafar.md` — чанк о замке на Pyrath и связи с тёмной стороной.

Поиск корректно вернул документы по разным аспектам сущности (биография, гибель, локации), что подтверждает качество индексации.

# Задание 4. Реализация RAG-бота с техниками промптинга

**Цель:** Создать работающего RAG-бота, который ищет информацию в векторной базе, формирует ответы с опорой на найденные данные и применяет техники Few-shot и Chain-of-Thought.

## Выбор LLM-провайдера

Для генерации ответов выбран **Groq API** с моделью **Llama 3 70B** (или эквивалентной 120B+ моделью).

| Критерий | Обоснование |
| :--- | :--- |
| **Скорость** | Сверхбыстрый инференс (до 800+ токенов/сек) через специализированные ASIC-чипы Groq. |
| **Качество** | Модели 70B+ параметров демонстрируют качество, сопоставимое с GPT-3.5/GPT-4, при значительно более низкой стоимости. |
| **Стоимость** | Оплата за токены, но цены на Groq значительно ниже OpenAI — экономия для корпоративного саппорта. |
| **Безопасность** | Данные передаются по HTTPS, но не хранятся на стороне провайдера (zero-data-retention policy). |

## Архитектура пайплайна

Скрипт `scripts/rag_bot.py` реализует модульный пайплайн по схеме:

```
Запрос пользователя → Эмбеддинг (all-MiniLM-L6-v2) → FAISS Retrieval (top-4)
    ↓
PromptTemplate (Few-shot + CoT + System) → ChatGroq → StrOutputParser
    ↓
Ответ с цитатами [1], [2] ... или "Я не знаю"
```

### Компоненты (Pearls vs Glass)

| Компонент | Реализация | Почему |
| :--- | :--- | :--- |
| **Retriever** | `FAISS.as_retriever(search_kwargs={"k": 4})` | Нативный LangChain-метод, 4 чанка — баланс между полнотой контекста и длиной промпта. |
| **LLM** | `ChatGroq` из `langchain_groq` | Актуальный пакет, нативная интеграция с LangChain, нет deprecated-обёрток. |
| **Цепочка** | LCEL: `prompt \| llm \| parser` | Прозрачная композиция вместо монолитных обёрток `RetrievalQA`. Легко добавить пост-фильтрацию или логирование. |
| **Парсер** | `StrOutputParser()` | Стандартный парсер для извлечения текста ответа без мета-информации. |

## Промпт-инжиниринг

### Few-shot prompting

В системный промпт встроены 2 примера из нашей обфусцированной базы:
- Вопрос о персонаже **Xarn Velgor** (Darth Vader) → демонстрация формата ответа с цитатой `[1]`.
- Вопрос о технологии **Synth Flux** (the Force) → демонстрация лаконичного ответа по документу.

Примеры используют реальные обфусцированные термины, что «задаёт тон» модели и снижает риск выдачи неструктурированного текста.

### Chain-of-Thought (CoT)

В промпте жёстко задан формат:

```
Рассуждение:
1. ...
2. ...
3. ...
4. ...
5. ...

Ответ: ...
```

CoT заставляет LLM декомпозировать задачу: определить сущность → найти релевантные документы → извлечь факты → сформулировать ответ → проверить достаточность. Это существенно снижает частоту галлюцинаций.

### Защита от галлюцинаций

Два уровня защиты:

1. **Pre-generation (retrieval layer):** если `retriever.invoke()` вернул пустой список документов, бот отвечает «Я не знаю» **до вызова LLM**. Это экономит токены и гарантирует, что модель не сможет «придумать» ответ.
2. **In-prompt guard:** системное сообщение содержит жёсткое правило: «Если документы не содержат ответа — честно скажи "Я не знаю"».

## Консольный интерфейс (REPL)

Бот запускается командой:
```bash
python scripts/rag_bot.py
```

- Загружает `.env` (переменные `GROQ_API_KEY`, `GROQ_MODEL_NAME`).
- Загружает FAISS-индекс из `vector_index/`.
- Работает в цикле `while True`: пользователь вводит вопрос, бот отвечает.
- Выход по командам: `exit`, `quit`, `выход`.

## Пример диалога (успешный ответ)

```
Вопрос: Кто такой Xarn Velgor и как он погиб?

Рассуждение:
1. Ключевая сущность — Xarn Velgor.
2. Документы [1] и [2] содержат информацию о Xarn Velgor.
3. Факты: родился Aeron Dawnstrider, пал к тёмной стороне, стал Void Disciple.
4. Погиб на Sylvara, уничтожив Overlord Malgrath.
5. Достаточно фактов: да.

Ответ: Xarn Velgor, родившийся Aeron Dawnstrider, был Void Disciple и Верховным командующим Dominion Ascendancy. Он погиб на луне Sylvara, уничтожив Overlord Malgrath и принеся себя в жертву. [1][2]
```

## Пример отказа («Я не знаю»)

```
Вопрос: Какой любимый цвет у Koda Veth?

Рассуждение:
1. Ключевая сущность — Koda Veth (Boba Fett).
2. В найденных документах нет информации о личных предпочтениях или любимом цвете.
3. Достаточно фактов: нет.

Ответ: Я не знаю
```

Таким образом, бот корректно отвечает на вопросы из базы знаний и честно признаётся в незнании, когда информация отсутствует.
