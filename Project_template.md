# Отчет по исследованию моделей и инфраструктуры (Задание 1)

**Контекст задачи:** Разработка RAG-бота для корпоративной базы знаний QuantumForge Software. 
**Ограничения:** Наличие конфиденциальной документации, регулярные SOC 2 аудиты, текущая инфраструктура на базе AWS (EKS + RDS).

## 1. Сравнение LLM-моделей

| Критерий | Локальные LLM (Hugging Face: Llama 3 8B, Qwen) | Облачные API (OpenAI, YandexGPT) |
| :--- | :--- | :--- |
| **Качество ответов** | Достаточное для RAG при качественном промптинге и CoT. | SOTA-качество, глубокое понимание контекста, низкий уровень галлюцинаций. |
| **Скорость (TTFT / TPS)** | Зависит от железа. Без GPU инференс неприемлемо медленный. | Высокая и предсказуемая, масштабируется на стороне провайдера. |
| **Стоимость (TCO)** | Высокий CapEx (закупка/аренда GPU). Нулевой OpEx за токены. | Оплата за токены (OpEx). При 18k документах и активном использовании может быстро расти. |
| **Развертывание** | Сложное (требуется настройка vLLM/Ollama, оркестрация). | Простое (REST API). |
| **Безопасность (SOC 2)** | **Идеально.** Данные не покидают закрытый контур компании. | Риски утечки. Требуются строгие NDA и Enterprise-тарифы с zero-data-retention. |

## 2. Сравнение моделей эмбеддингов

| Критерий | Локальные (Sentence-Transformers, BGE-m3) | Облачные (OpenAI text-embedding-3-small) |
| :--- | :--- | :--- |
| **Скорость индексации** | Высокая, особенно при использовании батчинга на CPU/GPU. | Ограничена rate limits API и сетевыми задержками. |
| **Качество поиска** | Отличное. Модели семейства BGE часто превосходят API-решения на задачах retrieval. | Высокое "из коробки", большое окно контекста. |
| **Стоимость** | Бесплатно. Крутится на существующих CPU-мощностях. | Оплата за каждый токен. Переиндексация базы требует повторных затрат. |

## 3. Сравнение векторных баз

| Критерий | FAISS | ChromaDB |
| :--- | :--- | :--- |
| **Тип решения** | Библиотека для поиска (C++), построение HNSW графов. | Полноценная векторная СУБД (DuckDB/SQLite под капотом). |
| **Скорость** | Максимально высокая производительность поиска. | Немного уступает FAISS на сыром поиске, но не критично для 18k документов. |
| **Внедрение и поддержка** | Требует написания собственной обертки для хранения метаданных и сериализации индексов на диск. | Прозрачная интеграция с LangChain. Сама управляет эмбеддингами, метаданными и персистенцией. |
| **Стоимость владения** | Минимальная (in-memory). | Низкая (легко поднимается в Docker-контейнере). |

*Примечание: для текущего MVP и быстрой проверки гипотез целесообразно использовать FAISS, так как он не требует подъема отдельного сервиса и легко упаковывается в минимальный Docker-образ.*

## 4. Рекомендуемые конфигурации инфраструктуры

**Вариант 1: MVP / Пилотный проект (Текущая реализация)**
* **Оборудование:** Локальная машина / базовая виртуалка (8 CPU cores, 16GB RAM).
* **Стек:** FAISS (in-memory) + локальные эмбеддинги (`all-MiniLM-L6-v2`) + внешняя LLM по API (OpenAI/YandexGPT).
* **Плюсы:** Максимальная скорость разработки, нулевые затраты на GPU, возможность быстро отладить RAG-пайплайн.

**Вариант 2: On-Premise Production (Полная изоляция для SOC 2)**
* **Оборудование:** Выделенный сервер (16 CPU cores, 32GB RAM, 1x GPU уровня RTX 4090 / A10G).
* **Стек:** ChromaDB + локальные эмбеддинги (BGE-m3) + локальная LLM (Llama 3 8B через vLLM).
* **Плюсы:** Полный контроль над данными, отсутствие OpEx за генерацию, соблюдение всех security-политик.

**Вариант 3: Cloud-Native Production (Интеграция в контур QuantumForge)**
* **Оборудование:** AWS EKS кластер (Node pools с GPU-инстансами `g5.xlarge` для LLM-воркеров).
* **Стек:** Qdrant/Milvus (как StatefulSet) + локальные эмбеддинги + локальная LLM.
* **Плюсы:** Нативная интеграция в текущую инфраструктуру компании, отказоустойчивость, автомасштабирование подов при пиковых нагрузках саппорта.

### Итоговый выбор для текущего проекта 

Для реализации бота в рамках данного спринта я выбираю **Вариант 1 (MVP)** с использованием **встроенного FAISS, локальных эмбеддингов (HuggingFace) и API внешней LLM**. 

**Аргументация:**
1. FAISS позволяет собрать минимально рабочий Docker-контейнер без необходимости оркестрации нескольких сервисов (что идеально для сдачи работы).
2. Локальные эмбеддинги позволяют бесплатно экспериментировать с чанкованием и переиндексацией.
3. Внешняя LLM по API снимает необходимость в GPU для локальной разработки, позволяя сфокусироваться на архитектуре самого RAG (настройке поиска, Few-shot и CoT промптинга), а не на тюнинге инференса моделей. В дальнейшем, при переводе в Production (AWS EKS), модуль генерации ответа можно легко переключить на локальную модель без изменения логики поиска.
